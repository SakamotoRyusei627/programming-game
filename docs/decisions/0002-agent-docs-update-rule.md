# 0002-agent-docs-update-rule

## ステータス

- 採用

## 日付

- 2026-06-14

## 背景

- 実装のたびに無差別に `docs` を追記すると、長期的に古い情報や作業ログが蓄積しやすい
- AI エージェントが更新判断を誤ると、入口構造の信頼性が下がる

## 決定

- 実装後に、仕様・設計・コマンド・制約・注意点が変わった場合だけ `docs` を更新する
- 単なる作業ログや一時メモは `docs` に残さない
- 古くなった記述を見つけた場合は追記ではなく修正を優先する

## 理由

- 長期的に再利用できる知識だけを残せる
- 古い情報を追記で積み増すより、修正中心の方が入口文書の精度を保ちやすい
- `docs` が実装補助の知識ベースとして機能しやすくなる

## 代替案

- すべての作業で必ず `docs` を更新する案
- 採用しなかった理由:
  - 更新対象が増えすぎて保守負荷が高い
  - 作業ログと恒久知識が混ざりやすい

## 影響

- 実装者は「仕様・設計・コマンド・制約・注意点が変わったか」を確認する必要がある
- `docs` 更新時はテンプレート参照と既存記述の修正を優先する
- 運用が定着したら、各カテゴリの未確認項目を順次具体化していく
