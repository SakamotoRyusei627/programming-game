# 0003-task-driven-tdd-workflow

## ステータス

- 採用

## 日付

- 2026-06-14

## 背景

- AI エージェントへ実装指示を出すとき、粒度と進め方が毎回ぶれると実装品質が安定しにくい
- 特に、タスク分解、テスト先行、変更前後の確認手順を共通化したかった

## 決定

- 実装指示は原則 `TASKS.md` の未完了タスクで管理する
- タスク化前の検討や要確認事項は `NOTES.md` に整理する
- 実装中に見つけた改善候補のうち、今回スコープ外のものは `PROPOSALS.md` に提案として残す
- エージェントは親タスクを確認後、詳細タスクへ分解してから実装を開始する
- 実装は TDD で進める
- TDD の工程は `テスト実装 → テスト Red 確認 → 実装 → テスト Green 確認 → リファクタリング` に固定する
- 変更前後の確認は原則 `make check` を入口とし、実行コマンドは `Makefile` に集約する

## 理由

- タスク分解を先に行うことで、影響範囲とテスト観点を揃えやすい
- 検討メモと実行タスクを分離すると、進捗管理が崩れにくい
- スコープ外の改善点を提案として残すと、自律的に見送った論点を失わずに済む
- テスト先行と Red/Green/Refactor を明示すると、実装の完了条件が曖昧になりにくい
- コマンド入口を `Makefile` に寄せると、プロジェクトごとの差分を吸収しやすい

## 代替案

- 実装指示を会話だけで渡す案
- テストや確認コマンドを都度エージェント判断に任せる案
- 採用しなかった理由:
  - 実装フローがぶれやすい
  - プロジェクトごとの運用知識が文書化されにくい

## 影響

- 新しいプロジェクトへ持ち込むときは `TASKS.md`、`NOTES.md`、`PROPOSALS.md`、`Makefile` の雛形を先に調整する
- 実際のテストコマンドが判明したら `Makefile` のプレースホルダを更新する
- 実装フローが変わった場合は `AGENT_GUIDE.md` と関連 `docs` も更新する
