# Frontend Tests

## AI向け要約

- フロントエンドの単体・統合・UIテスト方針を整理する文書です。
- 現在は `node:test` のスモークテストと `next build` によるビルド検証を使っています。
- 画面変更やコンポーネント変更時の検証観点を揃えるために参照します。

## このドキュメントを読むべきとき

- UIテストを追加するとき
- コンポーネント変更の影響範囲を確認したいとき
- モック方針や描画確認方法を確認したいとき

## 関連ファイル

| 種別 | パス | 説明 |
|---|---|---|
| 実装 | `src/app` | Next.js App Router の画面実装 |
| 実装 | `src/components/programming-game.tsx` | ゲーム UI 本体 |
| 実装 | `src/lib/programming-game.ts` | 盤面とコマンド実行ロジック |
| テスト | `tests/project-structure.test.mjs` | 主要ファイル配置のスモークテスト |
| テスト | `tests/programming-game.test.mjs` | ゲームロジックの単体テスト |
| 関連docs | [docs/frontend/components.md](../frontend/components.md) | UI責務の整理 |

## 基本方針

- UIテストは利用者から見える振る舞いを優先する
- 実装詳細への過剰な依存を避ける
- 通信や状態管理のモック境界を明確にする

## 実装ルール

- テストランナー: `node:test`
- レンダリングヘルパー: 未導入
- モック方式: 未導入
- スナップショット利用方針: 未導入
- ロジック変更時は UI に埋め込まず `src/lib` の純粋関数から先に検証する
- 変更時はアクセシビリティ観点も確認する

## よくある変更パターン

### パターン1: コンポーネントテスト追加

- 手順:
  1. 利用者視点の期待値を整理する
  2. 必要最小限のモックでテストを書く

### パターン2: フォームテスト追加

- 手順:
  1. 入力とエラー表示の流れを確認する
  2. 送信成功・失敗の両方を検証する

## 実行コマンド

```bash
npm run test
npm run build
```
