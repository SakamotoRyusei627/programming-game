## 基本方針

- データアクセスの詳細を上位層へ漏らしすぎない
- 業務ロジックはサービス層側へ寄せる
- パフォーマンス要件がある場合だけ最適化を文書化する

## 実装ルール

- ORM / SQL方式: 要確認
- リポジトリ単位: 要確認
- クエリ命名: 要確認
- N+1 や eager/lazy 方針: 要確認
- 変更時はインデックスや実行計画影響も確認する

## よくある変更パターン

### パターン1: 検索条件追加

- 手順:

  2. DTO と API 影響を合わせて確認する

### パターン2: 永続化方式変更

- 手順:
  1. トランザクション境界を確認する
  2. テストとマイグレーション影響を確認する
