# Frontend Routing

## AI向け要約

- 画面遷移、URL設計、ページ責務を整理する文書です。
- 現在はトップページ `/` をゲーム一覧、`/games/[gameId]` をゲーム詳細として使います。
- 画面追加やURL変更時は、一覧導線と `generateStaticParams` の更新漏れを確認します。

## このドキュメントを読むべきとき

- 新しい画面を追加するとき
- URLパターンを変更するとき
- リダイレクトやガード条件を確認したいとき

## 関連ファイル

| 種別 | パス | 説明 |
|---|---|---|
| 実装 | `src/app/page.tsx` | ゲーム一覧トップ |
| 実装 | `src/app/games/[gameId]/page.tsx` | ゲーム別ページ |
| 実装 | `src/features/games/catalog.ts` | 公開ゲーム一覧とID解決 |
| テスト | `tests/game-catalog.test.mjs` | カタログ定義の最小確認 |
| 関連docs | [docs/frontend/components.md](./components.md) | 画面責務の確認 |

## 基本方針

- URLは利用者視点で安定させる
- 画面責務とレイアウト責務を分離する
- 認証や権限に関わる遷移は `auth` カテゴリも合わせて確認する

## 実装ルール

- ルート定義場所: `src/app`
- ネスト構成: 一覧は `/`、ゲーム本体は `/games/[gameId]`
- 動的セグメント命名: `gameId`
- 遷移ガード方式: 未使用
- 変更時はリンク切れとリダイレクト影響を確認する

## よくある変更パターン

### パターン1: 新規画面追加

- 手順:
  1. 既存URL設計との整合を確認する
  2. 画面責務とアクセス条件を整理する

### パターン2: URL変更

- 手順:
  1. 既存リンクやブックマーク影響を確認する
  2. 必要ならリダイレクト方針を定める

## 実行コマンド

```bash
make test-unit
make test-e2e
```
